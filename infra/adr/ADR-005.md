# ADR-005: 무상태 크론 실행 모델

- **상태**: accepted
- **날짜**: 2026-02-28
- **결정자**: Owner + Jarvis

## 맥락 (Context)
LLM 에이전트를 장시간 실행(stateful daemon)하면 컨텍스트 드리프트가 발생한다. 이전 대화의 오류나 편향이 누적되어 점진적으로 품질이 저하되고, 원인 추적이 어렵다. 7팀 48개 크론 태스크가 서로 오염 없이 독립 실행되어야 했다.

## 결정 (Decision)
매 크론 실행을 완전 무상태로 설계한다. ask-claude.sh가 실행될 때마다: (1) /tmp에 새 작업 디렉토리 생성, (2) context/{task}.md에서 정적 컨텍스트 로드, (3) results/{task}/에서 최근 3건 히스토리 로드, (4) RAG 검색으로 관련 지식 주입, (5) 실행 후 /tmp 작업 디렉토리 삭제. 세션 간 상태 전달은 오직 파일을 통해서만.

## 대안 (Alternatives)
- Stateful daemon(상주 프로세스): 컨텍스트 재구성 비용 없음 — 선택하지 않은 이유: 드리프트 원천 차단 불가. 메모리 누수, 프로세스 좀비화 리스크. Mac Mini 재시작 시 상태 복원 복잡.
- 세션 유지 모델(claude --resume): 이전 대화 이어가기 가능 — 선택하지 않은 이유: Discord 대화에는 적합하나 크론 태스크는 매번 깨끗한 시작이 더 안전. resume 시 이전 세션의 오류가 전파될 위험.

## 결과 (Consequences)
- 긍정: 드리프트 원천 차단. 태스크 간 오염 불가. 실패 시 재실행이 곧 복구(별도 롤백 불필요). 디버깅 용이(results/ 파일로 입출력 완전 추적).
- 부정: 매 실행마다 컨텍스트 재구성 비용(RAG 쿼리 + 파일 읽기). 장기 대화형 태스크에 부적합(Discord 봇은 별도로 --resume 사용).
- 중립: cleanup trap이 /tmp 작업 디렉토리를 자동 삭제. PID 파일로 중복 실행 방지. caffeinate로 Mac 슬립 방지.

## 관련 결정
- ADR-001: claude -p 기반 아키텍처 (실행 엔진)
- ADR-002: 4계층 토큰 격리 (/tmp 작업 디렉토리가 격리의 첫 번째 계층)
- ADR-003: 파일 기반 에이전트 간 통신 (무상태 간 정보 전달 수단)
