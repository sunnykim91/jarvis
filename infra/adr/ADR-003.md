# ADR-003: 파일 기반 에이전트 간 통신

- **상태**: accepted
- **날짜**: 2026-03-01
- **결정자**: Owner + Jarvis

## 맥락 (Context)
자비스 컴퍼니 7팀(인프라, 정보, 기록, 성장, 학습, 브랜드, 의회) 에이전트가 정보를 공유해야 한다. 정보팀이 수집한 트렌드를 성장팀이 참조하고, 의회팀(council-insight)이 전 팀 현황을 종합하는 구조가 필요했다.

## 결정 (Decision)
파일 시스템을 유일한 통신 채널로 사용한다.
- `state/context-bus.md`: 의회팀이 매일 23:00에 갱신하는 전 팀 공유 게시판. ask-claude.sh가 모든 태스크에 자동 주입.
- `results/{task}/YYYY-MM-DD_HHMMSS.md`: 각 태스크 실행 결과. 다른 태스크가 이전 결과를 참조 가능.
- `context/{task}.md`: 태스크별 정적 컨텍스트(역할 정의, 참조 데이터).
- `state/shared-inbox/`: 팀 간 비동기 메시지 교환 디렉토리.

## 대안 (Alternatives)
- Redis Pub/Sub: 실시간 이벤트 처리 가능 — 선택하지 않은 이유: 추가 인프라 의존성. Mac Mini에 Redis 데몬 상시 구동 부담. 크론 기반 아키텍처에 실시간성 불필요.
- SQLite 메시지 큐: 구조화된 쿼리 가능 — 선택하지 않은 이유: 에이전트(claude -p)가 SQL 쿼리보다 마크다운 파일 읽기에 최적화. 복잡도 대비 이점 미미.
- 직접 API 호출(에이전트 간): 동기적 연쇄 호출 — 선택하지 않은 이유: claude -p 프로세스 간 직접 통신 불가. 크론 스케줄이 순서를 보장하므로 불필요.

## 결과 (Consequences)
- 긍정: 인프라 의존성 제로(파일 시스템만 필요). 모든 통신이 마크다운 파일로 남아 감사 추적 자동. git diff로 변경 이력 추적 가능. 에이전트가 자연어로 직접 읽기 최적.
- 부정: 실시간 통신 불가(최소 크론 주기 3분). 동시 쓰기 시 충돌 가능(현재 크론 스케줄로 회피). 파일 정리(rotation) 필요.
- 중립: results/ 디렉토리는 7일 자동 삭제(ask-claude.sh의 RESULT_RETENTION). context-bus.md는 매일 덮어쓰기.
