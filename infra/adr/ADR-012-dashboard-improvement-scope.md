# ADR-012 — 대시보드 UI/UX 개선 범위 및 배포 전략

**상태:** accepted
**날짜:** 2026-03-31
**결정자:** 이사회 (8명 전원 합의)

---

## Context

Jarvis Board 대시보드에 다음 이슈가 동시에 발생:

1. **sysMetrics null 버그** — 시스템 메트릭 데이터가 null/undefined일 때 대시보드 일부 섹션이 깨지거나 빈 화면 표시
2. **차트 기초 미비** — 시계열 차트가 데이터 없을 때 에러 상태로 표시
3. **진행률 바 미구현** — 그룹 태스크의 자식 태스크 완료 현황을 시각적으로 확인 불가
4. **아카이브 / 빈 상태 UI 미구현** — 완료 태스크 정리, 데이터 없는 화면 안내 부재

이들은 모두 **같은 데이터 파이프라인**(sysMetrics → 정규화 → 차트/KPI)을 공유하므로, 개별 배포 시 같은 레이어를 반복 수정해야 하고 리그레션 위험이 높다.

---

## Decision

1. **sysMetrics null 처리는 즉시 버그 수정** — 반나절 수준, 대시보드 신뢰도 직결
2. **백엔드 aggregation API 선행 구축** — 차트·진행률 바·아카이브가 공유하는 집계 API를 먼저 정리
3. **null 처리 + aggregation API + 차트 기초를 같은 스프린트에 묶어 배포** — 배포 횟수 최소화
4. **진행률 바·아카이브·빈 상태 UI는 후순위** — aggregation API 완성 이후, 오픈소스 공개 전 polish sprint에서 일괄 처리

---

## Rationale

- **배포 안정성**: 3개 기능이 데이터 정규화 레이어를 공유. 부분 배포 시 3회 리그레션 위험 → 일괄 배포로 1회로 줄임
- **코어 기능 우선**: null 처리는 기존 사용자 경험 깨짐이므로 최우선. 차트·진행률은 부가 기능
- **과잉 엔지니어링 방지**: 아카이브·빈 상태 UI는 현 시점에서 ROI 낮음. 오픈소스 공개(1년 후) 전 polish에서 충분

---

## Tradeoff

| 항목 | 이점 | 비용 |
|------|------|------|
| 일괄 배포 | 리그레션 1회, 데이터 레이어 한 번만 수정 | 초기 배포 시점 약간 지연 |
| null 처리 먼저 분리 배포 | 빠른 버그 수정 | 이후 차트 작업 시 같은 레이어 재수정 |
| 후순위 항목 동결 | 리소스 집중, 핵심 먼저 | 오픈소스 첫인상 자산 구축 지연 |

**선택: 일괄 배포** — null 처리와 aggregation API 설계를 같은 스프린트에 묶되, null 처리는 코드상 먼저 완성하여 중간 배포도 가능하게 구성.

---

## Consequences

- `lib/metrics.ts`의 정규화 함수가 차트와 KPI 카드의 단일 데이터 소스가 됨
- `GET /api/dev-tasks` 응답에 `total_children` / `completed_children` 필드 추가됨
- 프론트엔드 `DevTask` 타입에 해당 필드 반영됨
- 후순위 항목(진행률 바, 아카이브, 빈 상태 UI)은 별도 태스크로 백로그 등록
