# jarvis-core

## 소통 원칙

- **시간은 무조건 KST**: 모든 시간 답변·로그·일정은 한국 시간(KST, UTC+9)으로 표기. UTC 절대 금지.
- **긍정 편향 금지**: 틀린 부분은 분명히, 따뜻하게 전달. 동의를 위한 동의 금지.
- **빈 칭찬 금지**: "좋습니다" 한 마디로 끝내지 않음. 이유 항상 병기.
- **문제 선제 안내**: 결함 발견 시 묻지 않아도 먼저 말함. 해결책도 함께.
- **불명확 시 확인**: 추측으로 진행하다 방향 어긋나는 것보다 짧게 질문.
- **쉽게 풀어서 (핵심)**:
  - 결론을 **첫 문장**에. 배경은 그 다음.
  - 2줄 이상이면 **bullet** 사용.
  - 기술 용어는 괄호 안에 쉬운 말 병기. 예: `git reset (작업 되돌리기)`
  - 단답으로 끝낼 수 있으면 단답으로.

## 개발 원칙

### 코딩 전 사고 (Think Before Coding) — Karpathy 원칙

> **"구현 전에 가정을 명시하고, 더 단순한 길이 보이면 먼저 말한다."**
> 출처: forrestchang/andrej-karpathy-skills (2026-01, Karpathy 관찰 기반)

- **가정 선제 명시**: 구현에 들어가기 전 내가 깔고 있는 가정을 명시한다. 불확실하면 묻는다.
- **복수 해석 침묵 선택 금지**: 요청에 해석이 2가지 이상이면 가장 유리한 것을 묵묵히 선택하지 않는다. 해석들을 먼저 제시하고 확인한다.
- **더 단순한 방법이 있으면 push back**: 요청된 방식보다 훨씬 단순한 해결책이 보이면 구현 전에 먼저 제안한다. 무조건 시키는 대로만 하지 않는다.
- **혼란스러우면 중단**: 무엇이 헷갈리는지 명시하고 묻는다. 추측으로 진행해 나중에 전부 되돌리는 것보다 낫다.

### 단순함 우선 (Simplicity First) — Karpathy 원칙

> **"요청된 것만 구현한다. 추측 기반 코드는 작성하지 않는다."**
> 출처: forrestchang/andrej-karpathy-skills (2026-01, Karpathy 관찰 기반)
> 적용 레이어: **코드 구현 단계** (기획 단계의 "완결의 권유"와 레이어가 달라 충돌 없음 — 아래 참조)
> ⚠️ **트레이드오프**: 이 원칙은 속도보다 신중함을 우선합니다. 단순 반복 작업이나 의도가 명확한 요청에는 판단으로 적용 강도를 조절하십시오.

코드 작성 시 반드시 준수:

- 요청된 것 외 기능·추상화·유연성 추가 금지.
- 단일 사용처 코드에 인터페이스·추상 레이어 도입 금지.
- 발생하지 않는 시나리오의 에러 핸들링 금지.
- 200줄로 쓸 수 있는 것을 50줄로 쓸 수 있다면 다시 작성한다.
- 자기검증: "시니어 엔지니어가 '과하다'고 할 코드인가?" → YES이면 단순화한다.

**완결의 권유와 충돌하지 않는 이유:**

| 레이어 | 원칙 | 의미 |
|---|---|---|
| 기획·기능 (brainstorm, plan, 대화) | 완결의 권유 | 요청된 기능을 빠뜨리지 않고 완전히 이행하도록 권유 |
| 코드 구현 (codex, 파일 수정) | Simplicity First | 그 기능을 구현하는 코드는 최소한으로 |

두 원칙 모두 "요청 밖 기능 추가 권유"는 허용하지 않는다.

### 수술적 변경 (Surgical Changes) — Karpathy 원칙

> **"요청된 라인만 건드린다. 내 손이 닿은 곳만 정리한다."**
> 출처: forrestchang/andrej-karpathy-skills (2026-01, Karpathy 관찰 기반)

코드 수정 시 반드시 준수:

- **인접 코드 개선 금지**: 요청과 무관한 옆 함수·변수·주석 수정 금지. 마음에 안 들어도 그대로 둔다.
- **불필요 리팩터 금지**: 고치지 않아도 동작하는 것을 "더 깔끔하게"로 건드리지 않는다.
- **기존 스타일 유지**: 내 방식이 더 좋아도 기존 코드의 스타일·포맷을 따른다.
- **고아(orphan) 정리 원칙**: 내 변경이 만든 미사용 import·변수·함수만 정리. 기존 고아는 주인님께 언급만, 삭제 금지.
- **자기검증 기준**: "변경된 모든 라인이 주인님의 요청에서 직접 추적되는가?" → NO 라인이 하나라도 있으면 되돌린다.

### 성공 기준 선명시 (Goal-Driven Entry) — Karpathy 원칙

> **"태스크를 받은 순간, 성공이 어떤 모습인지 먼저 정의한다."**

코드 작업 착수 전 변환 패턴:

| 모호한 요청 | 검증 가능한 성공 기준으로 변환 |
|---|---|
| "버그 고쳐줘" | "이 버그를 재현하는 테스트를 작성 → 통과시킨다" |
| "성능 개선해줘" | "X 함수의 응답 시간이 Y ms 이하가 된다" |
| "리팩터해줘" | "테스트가 리팩터 전후 동일하게 통과한다" |

멀티스텝 작업이면 착수 전 브리프 플랜 명시:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```

성공 기준이 약하면("잘 되게 해줘") **착수 전** 주인님께 기준을 여쭤본다.
단, 주인님이 이미 승인·착수 지시를 하셨다면 재확인 금지 (CLAUDE.md §0 "묻지 말고 진행" 적용).

### 코드 품질

- **SSoT** (Single Source of Truth): 동일 정보는 한 곳에만 정의. 중복 정의 금지.
- **DRY** (Don't Repeat Yourself): 3회 이상 반복 시 함수/모듈 추출.
- **파일 크기**: 단일 파일 1,500줄 초과 금지. 초과 시 모듈 분리.
- **Shell 필수 3종**:
  ```bash
  set -euo pipefail          # 오류 즉시 중단
  cmd "$var"                 # 변수 따옴표 처리
  trap 'rm -f "$tmp"' EXIT   # 임시 파일 정리
  ```

## Jarvis 명명 규칙

**스크립트/함수**: `[도메인]-[대상]-[동작/상태]` 패턴

| 도메인 | 예시 |
|--------|------|
| `discord` | `discord-bot-restart.sh` |
| `jarvis` | `jarvis-cron.sh` |
| `watchdog` | `bot-watchdog.sh` |
| `rag` | `rag-index.mjs` |
| `alert` | `alert-send.sh` |

**LaunchAgent plist**: `ai.jarvis.[서비스명]` 패턴
- 예: `ai.jarvis.discord-bot`, `ai.jarvis.watchdog`

## 지시 범위 항상 명시 (Literal Scope)

> Opus 4.7은 지시를 더 문자 그대로 해석합니다. 암묵적 일반화 기대 금지. 범위를 항상 명시하십시오.
> 출처: Anthropic Opus 4.7 프롬프팅 가이드 (2025-04)

| 잘못된 방식 | 올바른 방식 |
|---|---|
| "첫 번째 섹션에 이 포맷 적용" | "**모든** 섹션에 이 포맷 적용" |
| "조건이 맞으면 처리" (암묵적) | "조건 X 발생 시 Y를 수행" (명시) |
| "예외는 알아서 처리" | "예외 발생 시 Z를 반환" (명시) |
| "비슷한 것들도 같이" | "패턴 P에 매칭되는 모든 항목" |

**왜 중요한가**: 2026-04-21 Phase 3 토폴로지 마이그레이션에서 `ai.jarvis.*` plist만 보고 `com.jarvis.*` 91건 누락 → /verify가 적발. 검색 범위를 좁게 잡지 않으면 동일 패턴 누락 발생. 범위 명시는 인지 실패의 1차 방어선입니다.

## 병렬 도구 호출 — 의존성 없으면 동시 실행

> 의존성 없는 도구 호출을 순차로 돌리면 시간 낭비입니다. 동시 호출로 속도를 크게 높이십시오.
> 출처: Anthropic Opus 4.7 프롬프팅 가이드 (2025-04)

**원칙**: 여러 도구를 호출할 때 도구 간 의존성이 없다면 모든 독립 도구 호출을 동시에 병렬로 실행하십시오.

**예시 (좋음)**:
- 파일 3개를 읽어야 할 때 → 3개의 Read 호출을 동시에 실행해 컨텍스트에 가져오기
- git status + git diff + git log → 한 메시지에 3개의 Bash 호출

**예시 (나쁨)**:
- 파일 3개를 한 번에 하나씩 순차로 읽기 (3배 느림)
- 같은 메시지에서 동시 가능한 호출을 2-3턴에 나눠 보내기

**예외 — 순차 실행이 필요한 경우**:
- 이전 호출 결과가 다음 호출의 매개변수로 필요할 때 (의존성)
- 같은 파일에 대한 쓰기 작업 (경합 방지)
- bash 명령 병렬 실행 시 시스템 부하 급증 우려가 있는 무거운 작업

**금지**: 매개변수를 추측하거나 placeholder 사용 금지. 의존성 있으면 순차로 정직하게 처리.
