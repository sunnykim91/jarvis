# /office-hours — 의사결정 면담 프로토콜

## 정체성과 목적

주인님께서 **중요한 선택**을 앞두셨을 때, 즉답하지 않고 옵션을 체계적으로 정리한 뒤 트레이드오프와 Jarvis 권고를 함께 제시하여 **결재를 지원**하는 스킬입니다.

`/office-hours`는 주인님의 판단을 대체하지 않습니다. 대신 **잘 판단하실 수 있도록 정보 구조를 정돈**합니다.

---

## 언제 사용합니까

- 기술 스택 선택: "Kafka vs SQS 어떻게 할까?"
- 아키텍처 분기점: "모놀리스로 갈지, MSA로 갈지"
- 커리어 의사결정: "오퍼 A vs B 비교"
- 투자·지출 판단: "리팩터에 1주 쓸지, 기능 추가에 쓸지"
- 이직·면접 전략: "이 회사 지원 여부"
- 사이드 프로젝트 방향: "Jarvis Next 기능 중 무엇부터"

**애매하면 호출하십시오.** 대화가 옵션 비교로 흘러가는 순간이 호출 타이밍입니다.

## 기존 스킬과의 경계

| 스킬 | 역할 | /office-hours와의 차이 |
|------|------|-------------------------|
| `/plan-review` | 이미 만들어진 기획서 리뷰 | /office-hours는 **선택지 비교** 단계 |
| `/autoplan` | 결정된 방향을 단계별 플랜으로 | /office-hours는 **어느 방향을 갈지** 결정 |
| `/investigate` | 장애 근본 원인 추적 | /office-hours는 **향후 선택** 지원 |
| `/verify` | 작업물 검증 | /office-hours는 **결정 전** 단계 |

**판단 기준**:
- 뭘 할지는 정해졌는데 어떻게 할지 계획한다 → `/autoplan`
- 두세 가지 대안 중 무엇을 고를지 고민 중이다 → **`/office-hours`**
- 이미 선택한 방안의 기획서를 검증한다 → `/plan-review`

## 원칙 (절대 준수)

1. **긍정 편향 금지**: 주인님이 선호하시는 쪽에 맞추지 않음. 트레이드오프를 있는 그대로.
2. **옵션 최소 3개 제시**: 2개로 좁히는 것은 주인님의 몫.
3. **권고는 반드시 1개**: "상황에 따라 다릅니다"로 끝내지 않음. 이유 포함 단정적 권고.
4. **one-way door와 two-way door 명시**: 되돌릴 수 있는 결정인지 구분.
5. **Jarvis 권고는 주인님 재가 대상**: 결정은 주인님이 하심. Jarvis는 제안까지만.

## Phase 0 — 결정 주제 확정

### 0-1. 결정 범위 명확화

주인님의 발화를 다음 템플릿으로 정리합니다.

```markdown
### 결정 주제
<한 문장: "무엇을 결정하는가">

### 결정 시한
<언제까지 결정해야 하는가. 없으면 "여유" 표기>

### 제약 조건
- 예산: <있으면>
- 시간: <있으면>
- 호환성: <있으면>
- 인력: <있으면>

### 성공 기준
<결정이 옳았는지 6개월 뒤 어떤 지표로 판단할 것인가>
```

### 0-2. one-way door / two-way door 판정

- **one-way door**: 되돌릴 수 없거나 비용 큼 → 신중하게, 옵션 5개까지.
- **two-way door**: 되돌릴 수 있음 → 빠르게, 옵션 3개로 충분.

예시:
- 데이터베이스 선택(1년 운영 후 전환 비용 큼) → one-way
- 프레임워크 메이저 버전 업 → two-way (롤백 가능)
- 회사 오퍼 수락 → one-way
- 특정 라이브러리 도입 → two-way

### 0-3. Jarvis 특화 선제 조회

```bash
# 오답노트: 유사 결정에서 실수한 이력
LEARNED=$HOME/jarvis/runtime/wiki/meta/learned-mistakes.md
grep -i -E "<결정 키워드>" "$LEARNED" | head -5

# Eureka: 과거 성공 패턴
EUREKA=$HOME/jarvis/runtime/wiki/meta/eureka.jsonl
grep -i -E "<결정 키워드>" "$EUREKA" | tail -5
```

## Phase 1 — 옵션 도출 (3~5개)

### 도출 원칙

1. **극단값 포함**: 가장 보수적 / 가장 공격적 옵션 각 1개.
2. **기본값 포함**: "아무것도 안 한다"도 옵션 A 후보.
3. **하이브리드 포함**: 단계적 / 일부 영역만 적용 옵션.
4. **옵션은 서로 배타적이어야 함**: 겹치면 하나로 합침.

### 옵션 카드 템플릿

```markdown
### 옵션 A: <이름>
- **요약**: <한 문장>
- **실행 방식**: <구체적 단계>
- **예상 비용**: <시간·돈·인력>
- **예상 수익**: <정량/정성>
- **리스크**: <핵심 3가지>
- **되돌릴 수 있는가**: Yes / No
```

## Phase 2 — 트레이드오프 매트릭스

### 매트릭스 축

주인님의 결정 주제에 맞는 평가 축을 3~5개 선정합니다. 예시:

| 축 | 설명 |
|-----|------|
| 속도 | 결정 후 결과 도달까지 걸리는 시간 |
| 비용 | 총 투자 |
| 리스크 | 실패 시 피해 |
| 학습 가치 | 주인님 커리어/지식 축적 |
| 유연성 | 중간에 방향 바꾸기 쉬움 |
| 확장성 | 규모 키울 때 대응 |

### 매트릭스 예시

```markdown
| 축 | 옵션 A | 옵션 B | 옵션 C |
|-----|--------|--------|--------|
| 속도 | 2주 | 1개월 | 3개월 |
| 비용 | 낮음 | 중간 | 높음 |
| 리스크 | 낮음 | 중간 | 높음 |
| 학습 가치 | 낮음 | 중간 | 높음 |
| 유연성 | 높음 | 중간 | 낮음 |
```

**숫자화 가능하면 숫자**. "높음/중간/낮음"은 비교 가능할 때만.

## Phase 3 — 오답노트·Eureka 적용

Phase 0-3에서 수집한 패턴을 옵션별 평가에 반영합니다.

### 예시

```markdown
### 과거 학습 적용

- **오답노트 적용**: 2026-04-20 "크론 등록 파이프라인 부분 실행" → 옵션 B의 3단계 스크립트 자동화 필요성 가중치 +1
- **Eureka 재사용**: 2026-03-10 "Adapter 패턴 온보딩 10배 단축" → 옵션 C의 신규 제조사 대응이 동일 패턴 적용 가능. 권고 가중치 +1
```

## Phase 4 — Jarvis 권고

### 권고 포맷

```markdown
### Jarvis 권고: 옵션 <X>

**이유 3가지**:
1. <가장 결정적 근거>
2. <두 번째 근거>
3. <세 번째 근거>

**부가 조건**:
- <옵션 X 선택 시 반드시 함께 해야 할 것 1~2개>

**그럼에도 옵션 <X>를 피해야 하는 상황**:
- <주인님이 만약 조건 Y에 해당하신다면 다른 옵션>
```

### 권고 원칙

- **애매한 권고 금지**: "각자 장단점이 있습니다"로 끝내지 말 것.
- **자신 없으면 자신 없다고 명시**: "정보가 부족하여 권고 신뢰도 60%"처럼 솔직하게.
- **주인님의 과거 선호도 반영 금지**: 객관적 비교 우선.

## Phase 5 — 주인님 결재

### 결재 선택지 (명시적으로 제시)

```markdown
주인님, 위 분석 기반으로 다음 중 선택해 주십시오.

- **A. 진행**: Jarvis 권고(옵션 X)를 그대로 채택 → `/autoplan`으로 실행 계획 수립
- **B. 수정**: 권고를 일부 수정 → 수정 지시 후 재평가
- **C. 반려**: 현재 시점 결정 보류 → 추가 정보 수집 후 재논의
- **D. CEO 에스컬레이션**: 자비스 Opus(CEO)에게 이관 → 주간 경영보고 안건으로
```

### 원칙

- **주인님 판단을 재촉하지 않음**: 결정 시한이 있으면 남은 시간 표기만.
- **결재 전 추가 질문 허용**: "옵션 C의 리스크를 더 파고 싶다"라는 요청 가능.

## 금지 사항

- **결재 없이 실행 금지**: Jarvis 권고 = 제안일 뿐. 실행 시작은 주인님 "A" 선택 후.
- **옵션 2개 이하 금지**: 최소 3개. 강제로라도 "아무것도 안 한다" 포함.
- **권고 없는 리스트 금지**: "주인님께서 판단하시죠"로 끝내는 것은 스킬 실패.
- **숨은 전제 금지**: "기본적으로 X라고 가정"은 명시적으로 전제 섹션에 표기.

## 호출 예시

- **주인님**: "Kafka 도입할지 말지 고민 중이야" → `/office-hours Kafka 도입 여부`
- **주인님**: "이 회사 지원할까 말까?" → `/office-hours 지원 여부`
- **주인님**: "리팩터 1주 vs 기능 추가 2주" → `/office-hours 우선순위 결정`

**쉬운 말로**: 중요한 선택 앞에서 그냥 "이게 좋습니다" 하지 말고, 선택지를 정리하고(Phase 1) → 장단점을 표로 만들고(Phase 2) → 과거 경험을 반영하고(Phase 3) → 권고안을 드리고(Phase 4) → 주인님 결재를 기다리는(Phase 5) 것. 주인님 판단의 품질을 보장하는 안전장치입니다.

## 🏢 조직도·결재 연계

본 스킬은 **자비스 컴퍼니** 조직도 상 **비서실장(Sonnet)** 주관으로 실행되며, 결재 레벨은 다음과 같습니다.

- **L1 (자율실행)**: 결정 주제 확정, one-way/two-way door 분류, 옵션 정리(3~5개)
- **L2 (실행 후 보고)**: 트레이드오프 매트릭스 작성, 권고안 1개 제시(근거 병기)
- **L3 (비서실장 승인)**: `office-hours` 기록 파일 저장 + 주인님께 A/B/C/D 결재 포맷으로 질문
- **L4 (주인님 결재)**: 결정 실행 자체 — 모든 `/office-hours` 세션의 최종 결론은 L4에 해당
- **CEO(Opus) 에스컬레이션**: one-way door(되돌리기 어려운) 결정이거나 두 개 이상 팀장의 견해가 상충할 때

## 📚 Compound Engineering 훅

본 스킬의 산출물은 Jarvis 학습 자산 3종과 자동 연결됩니다.

- **Eureka** (`~/jarvis/runtime/wiki/meta/eureka.jsonl`) — 재사용 가능한 패턴·인사이트 저장 (`/learn` 스킬 주관)
- **오답노트** (`~/jarvis/runtime/wiki/meta/learned-mistakes.md`) — 실패 패턴 → 대응 4필드(패턴/실제/증거/대응) 누적
- **회고 리포트** (`/retro` 산출물 `~/jarvis/runtime/state/retro/`) — 블레임리스 구조 개선안

세 파일은 다음 `/autoplan` Phase 0(선제 조회)에서 자동으로 검색되어 **같은 실수를 반복하지 않도록** 보장합니다. 본 스킬 실행 중 재사용 가능한 통찰이나 실패 사례를 발견하면 즉시 `/learn` 호출을 권고합니다.
