# 스펙 주도 개발 (SDD) 가이드

스펙 주도 개발(Spec-Driven Development) 프로세스를 시작합니다.
아래 단계에 따라 기능을 설계하고 구현을 진행해주세요.

## SDD 워크플로우

```
스펙 작성 → 리뷰 → 구현 → 인수 테스트 → 완료
```

---

## 1단계: Task 문서 작성

프로젝트 루트 또는 `docs/specs/` 디렉토리에 Task 문서를 작성합니다.

### 템플릿: `task-[기능명].md`

```markdown
# Task: [기능명]

## 개요
- **목적**: (이 기능이 왜 필요한가)
- **담당**: (구현 담당자)
- **우선순위**: P0/P1/P2
- **예상 범위**: S/M/L

## 유스케이스

### UC-1: [주요 시나리오]
- **선행 조건**: ...
- **정상 흐름**:
  1. 사용자가 ...
  2. 시스템이 ...
  3. 결과: ...
- **후행 조건**: ...

### UC-2: [보조 시나리오]
- ...

## 엣지케이스

| ID | 상황 | 예상 동작 | 비고 |
|----|------|----------|------|
| E-1 | 입력값 없음 | 에러 메시지 반환 | |
| E-2 | 네트워크 단절 | 재시도 3회 후 실패 | |
| E-3 | 동시 접근 | 락 획득 실패 시 대기 | |

## 제약 조건
- 성능: ...
- 호환성: ...
- 보안: ...

## 비기능 요구사항
- 로깅: ...
- 모니터링: ...
- 알림: ...
```

## 2단계: Development 문서 작성

### 템플릿: `dev-[기능명].md`

```markdown
# Development: [기능명]

## 아키텍처

### 시스템 구성도

(아래 Mermaid 다이어그램을 상황에 맞게 수정)

graph TD
    A[Client] --> B[API Gateway]
    B --> C[Service]
    C --> D[Database]
    C --> E[External API]

### 시퀀스 다이어그램

sequenceDiagram
    participant U as User
    participant S as Service
    participant D as DB
    U->>S: 요청
    S->>D: 조회
    D-->>S: 결과
    S-->>U: 응답

## 구현 계획

### 파일 변경 목록

| 파일 | 변경 유형 | 설명 |
|------|----------|------|
| `src/xxx.ts` | 신규 | 핵심 로직 |
| `src/yyy.ts` | 수정 | 인터페이스 추가 |
| `test/xxx.test.ts` | 신규 | 단위 테스트 |

### 데이터 흐름

1. 입력 → 검증 → 처리 → 저장 → 응답

### 의존성
- 기존 모듈: ...
- 외부 패키지: ...

## 주의사항
- 기존 코드와의 호환성
- 마이그레이션 필요 여부
- 롤백 계획
```

## 3단계: Feature 문서 (Single Source of Truth)

### 템플릿: `feature-[기능명].md`

```markdown
# Feature: [기능명]

> Single Source of Truth - 이 문서가 해당 기능의 최종 스펙입니다.

## 상태
- [ ] 스펙 확정
- [ ] 구현 완료
- [ ] 테스트 통과
- [ ] 리뷰 완료
- [ ] 배포 완료

## API / 인터페이스

### 입력
- 파라미터: ...
- 타입: ...
- 필수/선택: ...

### 출력
- 정상 응답: ...
- 에러 응답: ...

### 사용 예시

(코드 또는 커맨드 예시)

## 설정

| 키 | 기본값 | 설명 |
|----|--------|------|
| `config.key` | `value` | 설명 |

## 관련 문서
- Task: `task-[기능명].md`
- Dev: `dev-[기능명].md`
- 테스트: `test/[기능명].test.*`
```

## 4단계: 인수 테스트 케이스

### 템플릿: `acceptance-[기능명].md`

```markdown
# 인수 테스트: [기능명]

## 테스트 환경
- OS: macOS (Apple Silicon)
- Node.js: v22+
- 의존 서비스: ...

## 테스트 케이스

### TC-1: 정상 흐름 검증
- **전제**: ...
- **실행**: ...
- **기대 결과**: ...
- **결과**: PASS / FAIL

### TC-2: 엣지케이스 - 입력 없음
- **전제**: 입력값 미제공
- **실행**: ...
- **기대 결과**: 에러 메시지 "..."
- **결과**: PASS / FAIL

### TC-3: 엣지케이스 - 네트워크 에러
- **전제**: 외부 API 불가
- **실행**: ...
- **기대 결과**: 재시도 후 실패 로그
- **결과**: PASS / FAIL

## OpenClaw 특화 테스트

### TC-OC-1: Gateway 재시작 후 복구
- **전제**: Gateway 프로세스 재시작
- **실행**: 30초 대기 후 기능 호출
- **기대 결과**: 정상 동작
- **결과**: PASS / FAIL

### TC-OC-2: launchd 서비스 연동
- **전제**: 관련 LaunchAgent 로드됨
- **실행**: `launchctl list | grep ai.openclaw`
- **기대 결과**: 서비스 정상 등록
- **결과**: PASS / FAIL
```

## 실행 방법

1. **새 기능 시작**: "SDD로 [기능명] 시작해줘" 라고 요청
2. **기존 스펙 수정**: "task-xxx.md 엣지케이스 추가해줘"
3. **구현 진행**: "dev-xxx.md 기반으로 구현 시작해줘"
4. **인수 테스트**: "acceptance-xxx.md 실행해줘"
