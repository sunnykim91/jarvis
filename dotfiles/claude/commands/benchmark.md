# /benchmark — 벤치마크 실행 & 증거 기반 의사결정

주인님, 이 스킬은 **"어느 모델이 좋을까?", "이 프롬프트가 더 나을까?", "접근법 A vs B 중 무엇을 택해야 할까?"** 같은 질문을 **감이 아닌 측정**으로 답하기 위한 도구입니다. 한 줄 요약: **선택하기 전에 재보는 저울**.

원본 영감: `gstack/benchmark` (웹 성능 회귀 감지). 그 철학("Measure, don't guess")과 Phase 구조만 계승하고, 도메인은 모델/프롬프트/접근법 비교로 전면 재설계하였습니다. 외부 `gstack-*` 바이너리 의존은 전부 제거하였으며, 모든 로직은 본 스킬 본문에 내재합니다.

## 기존 스킬과의 경계 (반드시 확인)

주인님께서 다음 스킬들과 헷갈리실 수 있어 경계를 명시합니다.

| 스킬 | 목적 | 시점 | 산출물 |
|------|------|------|--------|
| **/verify** | 작업물 **사후** 검증 (7-Gate + Contrarian) | 이미 만들어진 결과물 | "이 작업물, 믿을 만한가?" |
| **/office-hours** | 의사결정 면담 (정성 + 정량 혼합) | 선택 직전 | 옵션/트레이드오프/권고 |
| **/review** | 코드 리뷰 (Dev + Reviewer) | PR/diff 단위 | 코드 품질 개선점 |
| **/benchmark** | **선택 전 정량 측정** (모델·접근법 자체를 저울에 올림) | 옵션이 2개 이상일 때 | 샘플별 수치 + 권고 |

**한 줄 판단 기준**: "이미 고른 걸 검증하는가"면 `/verify`. "숫자로 재봐야 결정 가능한가"면 `/benchmark`. "숫자 + 직관 섞어서 상의하고 싶은가"면 `/office-hours`.

## Phase 0 — Jarvis 특화 3종 선제 조회

측정을 시작하기 전, **과거의 실패와 통찰을 먼저 꺼내 봅니다**. 같은 실수를 반복하지 않기 위함입니다.

### 0-1. 오답노트 조회

```bash
if [ -f ~/jarvis/runtime/wiki/meta/learned-mistakes.md ]; then
  grep -iE "benchmark|벤치마크|측정|모델 비교|샘플" \
    ~/jarvis/runtime/wiki/meta/learned-mistakes.md || \
    echo "관련 오답노트 없음 — 처음 시도하는 도메인"
else
  echo "오답노트 파일 없음 — 초회 실행"
fi
```

발견된 유사 실수 패턴은 **주인님께 먼저 보고**한 뒤, 해당 실수를 이번 벤치마크에서 어떻게 피할지 Phase 1 목적 정의에 반영합니다.

### 0-2. Eureka 재사용

```bash
if [ -f ~/jarvis/runtime/wiki/meta/eureka.jsonl ]; then
  grep -iE "benchmark|모델|프롬프트|샘플|분산" \
    ~/jarvis/runtime/wiki/meta/eureka.jsonl | tail -5 || \
    echo "관련 Eureka 없음"
else
  mkdir -p ~/jarvis/runtime/wiki/meta
  touch ~/jarvis/runtime/wiki/meta/eureka.jsonl
  echo "Eureka 파일 초기화 완료"
fi
```

과거 벤치마크에서 얻은 "의외의 발견"(예: 특정 프롬프트 스타일이 지연시간을 반감시킨 사례)을 현재 설계에 반영합니다.

### 0-3. 조직도 포지셔닝

벤치마크는 **감사팀장**(Audit)과 **학습팀장**(Learning)의 공동 소관입니다.

- **감사팀장**: Iron Law 2 (거짓 완료 선언 금지) 집행 — "측정 안 하고 결론 내기" 차단
- **학습팀장**: Phase 5의 Eureka 등록과 오답노트 연계 담당 — Compound Engineering 체인

주인님께 보고 시 "감사팀장 시각: X / 학습팀장 시각: Y" 형식을 권장드립니다.

## Phase 1 — 벤치마크 목적 정의

**"무엇을 측정하는지 1문장으로 쓸 수 없으면, 측정하지 마십시오."**

### 1-1. 필수 질문 (주인님께 확인)

1. **비교 대상은 무엇입니까?** (예: Claude Opus 4.7 vs Sonnet 4.6, 프롬프트 A vs B, RAG 검색 k=5 vs k=10)
2. **성공 기준은 무엇입니까?** (예: 정확도 90% 이상, p50 응답 2초 이내, 비용 $0.05/요청 이하)
3. **허용 가능한 트레이드오프는 무엇입니까?** (예: "정확도 2% 낮아도 비용 절반이면 OK")

답이 모호하면 **여기서 멈추고 여쭙니다** (Iron Law 3). 추측으로 진행 금지.

### 1-2. 측정 축 선택

다음 4축 중 최소 2개 이상을 선택합니다.

| 축 | 단위 예시 | 주의점 |
|----|-----------|--------|
| **정확도** | 정답률 %, F1, Rubric 점수 | 채점 기준을 먼저 고정 |
| **속도** | p50 / p95 ms, 초당 처리량 | 첫 요청(cold) 제외 여부 명시 |
| **비용** | $/요청, 토큰 수 | 입출력 토큰 분리 집계 |
| **재현성** | 동일 입력 시 출력 분산 | 온도(temperature) 고정 필수 |

### 1-3. 목적 기록 (파일 생성 X, 변수에 저장)

```
목적: <1문장>
비교 대상: [A, B, C]
측정 축: [정확도, 속도, 비용]
성공 기준: <수치 목표>
트레이드오프 허용: <문구>
```

## Phase 2 — 샘플 세트 구성

**10~30개의 대표 케이스.** 너무 적으면 통계가 무의미하고, 너무 많으면 비용이 터집니다.

### 2-1. 케이스 구성 비율 (권장)

| 구성 | 비율 | 설명 |
|------|------|------|
| **대표(typical)** | 50~60% | 실제 프로덕션에서 자주 오는 정상 케이스 |
| **엣지(edge)** | 25~35% | 경계값, 긴 입력, 특수문자, 다국어 |
| **적대(adversarial)** | 10~15% | 일부러 실패를 유도하는 케이스 (재밍, 모호성) |

### 2-2. 샘플 수 가이드

| 측정 축 | 최소 샘플 수 |
|---------|--------------|
| 정확도만 | 20개 |
| 정확도 + 속도 | 15개 × 3회 반복 |
| 4축 전부 | 10개 × 3회 반복 |

### 2-3. 샘플 저장 형식

```jsonl
{"id":"s01","category":"typical","input":"...","expected":"...","rubric":"..."}
{"id":"s02","category":"edge","input":"...","expected":"...","rubric":"..."}
```

임시 경로: `~/jarvis/runtime/state/benchmarks/.samples-<slug>.jsonl` (최종 보고서에 경로 기록)

## Phase 3 — 실행 및 측정

**모든 비교 대상에 동일한 샘플을 줍니다.** 샘플 순서도 동일하게 유지 (순서 효과 제거).

### 3-1. 실행 루프 (의사코드)

```
for 대상 in [A, B, C]:
  for 샘플 in 샘플세트:
    for 반복 in [1, 2, 3]:
      start = now()
      결과 = 실행(대상, 샘플.input)
      duration = now() - start
      점수 = 채점(결과, 샘플.expected, 샘플.rubric)
      기록(대상, 샘플.id, 반복, duration, 점수, 토큰수, 비용)
```

### 3-2. 측정 시 반드시 고정해야 할 변수

- **temperature**: 재현성 측정이면 0, 그 외는 0.3 등으로 고정
- **max_tokens**: 모든 대상 동일
- **시스템 프롬프트**: 비교 대상 외 요소는 전부 동일
- **네트워크/시간대**: 가능하면 같은 세션 내에서 연속 실행

### 3-3. 채점 방식

| 유형 | 채점자 |
|------|--------|
| **객관식/추출** | 정규식 또는 완전 일치 스크립트 |
| **요약/생성** | Rubric 기반 LLM 채점 (별도 모델 권장) |
| **코드 생성** | 테스트 실행 + 단위 테스트 통과율 |

LLM 채점을 쓸 때는 **채점자 모델이 비교 대상과 겹치지 않도록** 분리합니다 (편향 제거).

## Phase 4 — 통계적 유의성

**1회성 시도로 권고를 결정하지 않습니다** (금지 사항).

### 4-1. 반복 수의 의미

| 반복 횟수 | 판단 가능 범위 |
|-----------|----------------|
| 1회 | 데모용. **권고 금지** |
| 3회 | 중앙값 + 분산 기초 |
| 5회 | IQR(사분위수 범위) 의미 있음 |
| 10회 이상 | 통계적 안정성 확보 |

### 4-2. 보고에 반드시 포함할 통계

- **중앙값(median)**: 평균 대신 중앙값 우선 (이상치 영향 제거)
- **IQR (Q3 - Q1)**: 분산의 직관적 지표
- **승률**: 샘플별 1대1 비교에서 각 대상이 이긴 비율
- **분산이 큰 샘플**: 별도 표시 (재설계 후보)

### 4-3. p-value는 요구하지 않음

벤치마크 목적상 **실용적 차이(practical significance)**가 중요하며, 통계적 유의성(p-value)까지 요구하지는 않습니다. 다만 IQR이 중앙값 차이보다 크면 **"유의미한 차이 없음"**으로 보고합니다.

### 4-4. 분산 계산 스니펫 (참고)

```bash
echo "$DATA" | jq -r '.[]' | sort -n | awk '
  { a[NR]=$1 }
  END {
    n=NR
    med = (n%2==1) ? a[int(n/2)+1] : (a[n/2]+a[n/2+1])/2
    q1  = a[int(n*0.25)+1]
    q3  = a[int(n*0.75)+1]
    printf "median=%.3f  IQR=%.3f  (Q1=%.3f, Q3=%.3f)\n", med, q3-q1, q1, q3
  }'
```

```markdown
| 대상 | 정확도(중앙값) | 속도 p50 (ms) | 비용/요청 ($) | 승률 | IQR(속도) |

주인님, **B 권고드립니다**. 이유:

2. 속도가 A 대비 2배 빠름, C 대비 3.5배
3. 비용은 A의 43%, C의 22%
4. 단, 적대 케이스에서 IQR이 크므로 프로덕션 투입 시 모니터링 필수
```

## 금지 사항 (Iron Laws 집행)

주인님, 다음은 **절대 해서는 안 되는** 행동입니다.

| 금지 행동 | 이유 | 해당 Iron Law |
|-----------|------|---------------|
| 외부 `gstack-*` 바이너리 호출 (예: `gstack-model-benchmark`) | 의존성 봉인 위반, 로직 블랙박스화 | 원칙상 내재 필수 |
| 측정 없이 "아마 A가 나을 것입니다"로 결론 | 추정을 확실로 위장 | **Iron Law 2** |
| 1회성 시도로 권고 결정 | 분산 무시, 우연을 법칙으로 착각 | **Iron Law 6** |
| 샘플을 비교 대상별로 다르게 구성 | 측정의 기본 전제 파괴 | 정확성 > 편의성 |
| 시크릿/PII가 샘플에 섞인 채 저장 | 로그 노출 위험 | **Iron Law 4** |
| "완료하였습니다"를 실제 실행 전 선언 | 거짓 보고 | **Iron Law 2** |

