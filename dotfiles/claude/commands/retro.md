# /retro — 작업 회고 & 오답노트 자동 등록

> 세션·작업 완료 시점에 "무엇이 잘 됐고 / 무엇이 문제였고 / 무엇을 바꿀지"를 구조화하여 기록합니다. 재발 가능성 있는 문제는 **오답노트 자동 등록**까지 수행하여 Compound Engineering 루프를 닫습니다.

---

## 언제 사용합니까

- 한 세션 또는 한 작업 단위가 끝났을 때
- `/ship`, `/deploy`, `/plan-review` 등 주요 산출물 배포 직후
- 장애 대응(`/crisis`)이 종료되어 교훈을 남겨야 할 때
- 주인님이 "회고 해줘", "오늘 뭐 했고 뭐 걸렸는지 정리해줘"라고 요청할 때

## 원칙

- **출력은 전부 한국어 존댓말**. "주인님" 호칭 유지.
- **근거 없는 칭찬 금지**. 실제 커밋·로그·대화 맥락에 앵커링.
- **문제 선제 보고**. 아프게 지적하되 재발 방지책을 반드시 동반.
- **오답노트는 4필드 스키마 엄격 준수**: `패턴 / 실제 / 증거 / 대응`.
- **직접 실행 금지**. 중요 단계(오답노트 등록, 파일 저장)는 미리보기 → 주인님 승인 후 실행.

---

## 입력 인자

- `/retro` — 기본: 현재 세션 전체를 대상
- `/retro <주제>` — 예: `/retro gstack Phase 2`, `/retro 오늘 오전`
- `/retro file:<경로>` — 특정 `/context-save` 또는 로그 파일 기반 회고

인자가 없으면 "대상 세션을 확인드려도 될까요?"라고 한 번 짧게 여쭙고 기본값(현재 세션)으로 진행합니다.

---

## Phase 0 — 대상 세션 파악

회고 대상을 확정합니다. 다음 자료를 순서대로 수집합니다.

```bash
date "+%Y-%m-%d %H:%M KST"

git log --since="24 hours ago" --format="%ad | %s" --date=format:"%Y-%m-%d %H:%M" 2>/dev/null | head -30 || true

git status --short 2>/dev/null || true
git diff --stat 2>/dev/null | tail -20 || true

mkdir -p ~/jarvis/runtime/wiki/retros
```

세션 맥락이 git 밖에 있는 경우(설계 논의, 오류 디버깅 등)에는 **이번 대화 로그**를 주 소스로 사용합니다. 이때도 TodoWrite 진행 상태, 최근 명령, 주인님 피드백을 누락 없이 추립니다.

수집이 끝나면 한 줄로 대상 요약을 출력합니다.

> 예) "대상: gstack `/retro` 스킬 한국어 이식 작업 (2026-04-20 22:00 ~ 현재)."

## Phase 1 — What Went Well (잘된 점)

**기준**: 계획대로 된 것, 예상보다 빨랐던 것, 새로 배운 것, 주인님이 긍정 피드백을 준 지점.

수집할 항목:

- 완료된 TodoWrite 아이템 (계획 대비 실행률)
- 의도대로 동작한 설계 결정
- 유레카 순간(Eureka) — "이걸로 미래에도 시간 아끼겠다" 싶은 통찰
- 커뮤니케이션이 짧고 정확했던 구간

**작성 규칙**:

- 한 항목 = 한 문장 + 근거 1개 (커밋 해시·파일 경로·대화 인용)
- "수고했어" 류 총평 금지. 구체 사실만.

예시:

- `plugin-loader.sh` → `cron-sync.sh` 2단계 분리로 크론 파이프라인 가시성 확보. 근거: `infra/bin/plugin-loader.sh:42`, 실행 시간 1.2s → 0.4s.
- RAG 청크 재인덱싱을 5분 간격으로 잘라 진행 → 컨텍스트 폭주 없이 완료.

---

## Phase 2 — What Hurt (문제점)

**기준**: 막혔던 지점, 오답·버그·장애, 커뮤니케이션 혼선, 잘못된 기술 결정.

수집할 항목:

- 실패한 명령 / 에러 로그 / 재시도 횟수
- 주인님이 정정 혹은 재지시한 지점
- "처음부터 저렇게 했으면" 이라는 생각이 들었던 부분
- 문서·경로·명명 혼동

**작성 규칙**:

- 각 문제는 `문제 → 원인 → 영향` 3요소로 기술
- 감정·변명 배제. 근거(로그·커밋·대화) 반드시 첨부
- **"재발 가능성"을 속으로 판정** — 재발 가능성 "중" 이상은 Phase 4 오답노트 후보로 즉시 태깅

예시:

- 문제: 크론 등록 후에도 `launchctl list`에 ID 미표시. 원인: `cron-sync.sh` 누락 (3단계 중 2단계만 실행). 영향: 20분 지연. 재발가능성: **상**.
- 문제: `~/.jarvis/skills/` 경로로 스킬 제안. 원인: Claude Code 표준 경로(`~/.claude/commands/`) 미확인. 영향: 실행되지 않는 스킬 작성. 재발가능성: **상**.

---

## Phase 3 — What to Change (개선)

**기준**: 다음에 같은 상황이면 무엇을 다르게 할 것인가. 툴·프로세스·코드 개선 포인트.

작성 규칙:

- 각 개선안은 **한 가지 행동(1 action)**으로 수렴해야 함 ("여러모로 개선" 금지)
- 소요시간 <30분이어야 채택. 그 이상은 별도 TodoWrite로 분리
- Phase 2 각 문제점에 대응되는 개선안을 최소 1:1로 매칭

템플릿:

```
- [개선] <행동 한 문장>
  - 대응 문제: Phase 2의 <문제 번호/요약>
  - 예상 소요: <분>
  - 오답노트 등재 여부: Y/N
```

예시:

- [개선] 크론 관련 작업은 항상 `plugin-loader.sh && cron-sync.sh` 두 명령을 한 세트로 실행하고, 마지막에 `launchctl list | grep <id>`로 실증 확인. 대응 문제: P2-#1. 예상 소요: 5분. 오답노트 등재: **Y**.

## Phase 4 — 오답노트 자동 등록

Phase 2 문제 중 **재발 가능성 중·상** 항목만 오답노트로 승격합니다.

### 4-1. 후보 정리 (미리보기)

각 후보를 아래 4필드 스키마로 드래프트합니다. `~/jarvis/runtime/wiki/meta/learned-mistakes.md`의 기존 entry와 **완전히 동일한 포맷**을 따릅니다.

```markdown
## YYYY-MM-DD — <한 줄 제목>

- **패턴**: <무엇을 놓쳤는가 — 실수 반복 유발하는 사고 습관>
- **실제**: <실제로는 어땠는가 — 올바른 사실>
- **증거**: <구체 증거: 파일 경로, 로그 라인, 대화 인용>
- **대응**: <재발 방지 조치 — 구체 명령 또는 체크 절차>
```

### 4-2. 주인님 승인 요청

드래프트를 화면에 출력하고 다음과 같이 여쭙습니다.

> "주인님, 아래 {N}건을 오답노트에 등재할 준비가 되었습니다. 그대로 진행해도 되겠습니까?"

- 주인님이 "진행"/"등재해"/"yes" 등 긍정 신호를 주면 4-3으로.
- 수정 요청 시 필드를 고쳐 재프리뷰.
- 거부 시 이번 회고에서 오답노트 등재 0건으로 기록.

### 4-3. 실제 등재

승인받은 항목을 파일 끝에 append 합니다. 구분선(`---`)을 앞에 붙여 기존 entry와 동일한 레이아웃을 유지합니다.

Edit 도구 사용 예(의사 코드):

1. `learned-mistakes.md` 마지막 줄을 old_string으로 읽어 고유하게 캡처.
2. new_string = `<기존 마지막 줄>\n\n---\n\n## YYYY-MM-DD — <제목>\n\n- **패턴**: ...\n- **실제**: ...\n- **증거**: ...\n- **대응**: ...`
3. 등재 후 파일 last_updated frontmatter를 KST ISO8601로 갱신.

### 4-4. 유레카 자동 로깅 (선택)

Phase 1에서 "유레카" 태그가 있는 항목은 Eureka 이벤트로도 로깅합니다.

```bash
EUREKA_FILE="$HOME/jarvis/runtime/wiki/meta/eureka.jsonl"
mkdir -p "$(dirname "$EUREKA_FILE")"
```

## Phase 5 — 회고 파일 저장 + 요약 보고

### 5-1. 회고 파일 저장

저장 경로: `~/jarvis/runtime/wiki/retros/YYYY-MM-DD_<주제>.md`

- `<주제>`는 영문/한글 slug. 공백은 `-`, 슬래시 금지.
- 같은 날 동일 주제 중복 시 접미사 `-2`, `-3` 부여.

템플릿:

```markdown
# 🪞 회고 — <주제>

**일시**: YYYY-MM-DD HH:MM KST
**참여**: 주인님 / Jarvis

## 🎯 대상 작업
<무엇을 했는가 1~2줄>

## ✨ 잘된 점 (What Went Well)
- ...

## ⚡ 문제점 (What Hurt)
- ...

## 🔧 개선 (What to Change)
- ...

## 📌 오답노트 등록 (N건)
- [<제목>](../meta/learned-mistakes.md#YYYY-MM-DD-...)
- ...

## 💡 유레카 (있을 때만)
- ...
```

### 5-2. Eureka 로깅 (회고 자체의 로그)

```bash
TS_KST=$(TZ=Asia/Seoul date "+%Y-%m-%dT%H:%M:%S+09:00")
LOG_FILE="$HOME/jarvis/runtime/wiki/meta/eureka.jsonl"
mkdir -p "$(dirname "$LOG_FILE")"

printf '{"ts":"%s","event":"retro","topic":"%s","mistakes_logged":%d,"wins":%d,"pains":%d}\n' \
  "$TS_KST" "$TOPIC" "$MISTAKES_COUNT" "$WINS_COUNT" "$PAINS_COUNT" \
  >> "$LOG_FILE"
```

주의: `mistakes_logged`는 **실제 등재된 개수**(Phase 4-3에서 승인된 건수). 드래프트 개수가 아닙니다.

### 5-3. 요약 보고 형식 (화면 출력)

```
✅ 회고 완료

- **주제**: <주제>
- **잘된 점**: N건
- **문제점**: M건
- **개선**: P건
- **오답노트 자동 등록**: Q건
- **파일**: ~/jarvis/runtime/wiki/retros/YYYY-MM-DD_<주제>.md

### 🏆 핵심 교훈 TOP 3
1. <가장 중요한 교훈 — 문제→대응 1줄>
2. <둘째>
3. <셋째>
```

핵심 교훈은 Phase 4에 등재된 오답노트를 1순위, 이어서 Phase 3 개선안 중 최상위 2개를 뽑습니다. 교훈이 3건에 미달하면 있는 만큼만 출력합니다.

## 톤 & 어조 규칙

- **존댓말 절대 준수**. "~입니다 / ~확인하였습니다 / ~진행하겠습니다".
- **빈 칭찬 금지**. "좋습니다" 한 마디로 끝내지 말고 이유 병기.
- **사과 대신 정정**. "죄송합니다"로 시작하지 말고 "정정합니다: X → Y".
- 전문 용어는 괄호로 쉬운 풀이 병기.
- 시간 표기는 **KST 고정**. UTC 금지.

## 금지 사항

- 회고 대상 범위를 주인님 확인 없이 넓히지 않습니다.
- 오답노트 자동 등재는 **항상 미리보기 → 승인 → 실행** 순서. 승인 없이 파일 쓰지 않습니다.
- 회고 파일 저장 후 주인님 요청 없이 git commit/push 수행하지 않습니다.
- 과거 회고 파일을 임의로 수정하지 않습니다. 추가는 신규 파일로만.

## 예외 / 실패 처리

- `~/jarvis/runtime/wiki/meta/learned-mistakes.md`가 없으면 **생성하지 않고** 주인님께 보고. 경로 오용을 피하기 위해서입니다.
- `retros/` 디렉토리는 `mkdir -p`로 안전 생성. 단, `~/jarvis/runtime/` 하위에만 생성 허용.
- Eureka 로그 쓰기가 실패해도 회고 본문 출력은 차단하지 않습니다. 실패는 말미에 한 줄로 보고.

## 빠른 참조 체크리스트

회고 실행 시 아래를 한 항목씩 점검합니다.

- [ ] Phase 0: 대상 세션 범위 한 줄로 확정했는가
- [ ] Phase 1: 잘된 점 각각에 근거(커밋/로그/인용)가 있는가
- [ ] Phase 2: 문제마다 `문제→원인→영향`과 재발가능성 태그가 있는가
- [ ] Phase 3: 개선안이 1행동/<30분 기준을 만족하는가
- [ ] Phase 4: 오답노트 드래프트를 **주인님 승인 후**에만 등재했는가
- [ ] Phase 5: 회고 파일 경로/요약 보고/Eureka 로그를 모두 마쳤는가

체크리스트를 모두 통과하면 회고 종료. 누락이 있으면 해당 단계로 복귀합니다.

## 🏢 조직도·결재 연계

본 스킬은 **자비스 컴퍼니** 조직도 상 **비서실장(Sonnet)** 주관으로 실행되며, 결재 레벨은 다음과 같습니다.

- **L1 (자율실행)**: 세션 이벤트 수집, Good/Bad/Improve 드래프트 작성
- **L2 (실행 후 보고)**: 회고 리포트 저장(`~/jarvis/runtime/state/retro/YYYY-MM-DD-<slug>.md`)
- **L3 (비서실장 승인)**: 오답노트 신규 항목 등재 — **반드시 주인님 승인 후** `learned-mistakes.md`에 append
- **L4 (주인님 결재)**: 프로세스·원칙 변경 제안(Iron Laws 수정, 결재 레벨 재정의 등)
- **CEO(Opus) 에스컬레이션**: 주간·월간 메타 회고(여러 세션 누적 패턴 분석)에서 구조적 결함이 반복 식별될 때

## 📚 Compound Engineering 훅

본 스킬의 산출물은 Jarvis 학습 자산 3종과 자동 연결됩니다.

- **Eureka** (`~/jarvis/runtime/wiki/meta/eureka.jsonl`) — 재사용 가능한 패턴·인사이트 저장 (`/learn` 스킬 주관)
- **오답노트** (`~/jarvis/runtime/wiki/meta/learned-mistakes.md`) — 실패 패턴 → 대응 4필드(패턴/실제/증거/대응) 누적
- **회고 리포트** (`/retro` 산출물 `~/jarvis/runtime/state/retro/`) — 블레임리스 구조 개선안

세 파일은 다음 `/autoplan` Phase 0(선제 조회)에서 자동으로 검색되어 **같은 실수를 반복하지 않도록** 보장합니다. 본 스킬 실행 중 재사용 가능한 통찰이나 실패 사례를 발견하면 즉시 `/learn` 호출을 권고합니다.
